# STAGE 1 — SPECIFY

Read AGENTS.md. Write specs/base.md — a specification for "Fighting Game", a turn-based
1v1 battle arena with server-authoritative turn resolution.

Cover:
- Fighter model: hp, ki, atk, def, spd, plus per-match "ascended" and "ascend_used" flags.
- Two starter fighters, Kaito and Vega, with distinct stat spreads (one glass-cannon, one tanky).
- Moves and exact rules for each: Strike (0 ki, small dmg), Ki Blast (small ki, medium dmg),
  Surge Beam (large ki, high dmg), Charge (restore ki, no attack), Guard (halve next incoming
  damage + restore small ki), Ascend (once/match: spend ki, permanently buff atk & spd).
- Damage formula using atk vs def with a small random spread, how guard and ascend modify it,
  and how ki costs/regen work. Make the numbers concrete.
- A single resolve-turn operation: input = both fighters' state + player action + opponent action;
  output = new state + a battle-log line. Opponent for the base game just picks a legal random move.
- REST endpoints: start a match, submit a turn, get match state. Define request/response JSON.
- Win/lose condition and how the battle log is represented.
- Acceptance criteria as a checklist a reviewer can verify, and a test plan (unit tests for the
  damage/ki rules, endpoint tests for the API).

Also create the script/ files (setup, server, test, lint) so the stack in AGENTS.md runs.
Do not implement game logic yet — spec and scripts only.


# STAGE 2 — REVIEW

Act as a skeptical technical reviewer of specs/base.md. Check: are the damage and ki numbers
balanced enough that no single move dominates and matches actually end? Is every move's edge case
defined (guarding with 0 ki, beam with insufficient ki, ascending twice)? Is the resolve-turn
contract unambiguous? Are the acceptance criteria testable and the test plan adequate?
List concrete problems and, for each, edit specs/base.md to fix it. Do not implement game code.

# STAGE 1 — SPECIFY

Read AGENTS.md and specs/base.md. Write specs/extension.md covering TWO additions:

(A) AI opponent that replaces the random move-picker:
   - A heuristic policy first: charge when ki is low; use Surge Beam when it can KO or when ki is
     high; Ki Blast as the default poke; Guard when a big incoming hit is likely; Ascend at a
     defined ki/HP threshold.
   - Then a shallow expectimax search (depth ~2-3) over the resolution function, averaging over the
     random damage spread, with a stated position-evaluation function (hp/ki differential, tempo).
   - A difficulty setting that selects heuristic vs. search.

(B) Persistent single-elimination tournament:
   - SQLAlchemy models for Fighter, Tournament, Match (with bracket position + result).
   - Endpoints: create a tournament from a roster, play/advance the next match (AI vs AI or
     player vs AI), fetch bracket + standings. Results persist across server restarts (SQLite).
   - A bracket view in the UI showing rounds, winners, and the champion.

Include acceptance criteria and a test plan for both (unit tests for the AI policy on crafted
board states; DB tests for bracket creation, advancement, and persistence).
Do not implement yet.


# STAGE 2 — REVIEW

Review specs/extension.md as a technical reviewer. Is the expectimax evaluation function well
defined and cheap enough to run per turn? Does the AI ever get stuck in a charge/guard stalemate,
and is that prevented? Are the DB models normalized, and is bracket advancement unambiguous for
byes and odd rosters? Are persistence guarantees actually testable? Fix each issue directly in
specs/extension.md. Do not implement.
